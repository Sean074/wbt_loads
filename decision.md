## 1. Input File Formats

The spec names all required inputs but never defines the file format for any of them.
A format must be chosen for each before the corresponding parser module can be coded.

### 1a. Aerodynamic database (Cn, Cm, Cc tables)
Options: one CSV per surface per variable; HDF5 multi-table; JSON; CPACS XML.

**Direction:** [x] Two-tier CSV structure per lifting surface. Baseline tables
(`aero_<surface>_<config_tag>.csv`) store full strip coefficients indexed by
`(y_m, alpha_deg, beta_deg, mach_nd)` for each power/flap configuration.
Increment tables (`aero_incr_<surface>_<control>.csv`) store delta coefficients
for each control surface, also indexed by `(y_m, alpha_deg, beta_deg, mach_nd,
deflection_deg)`. Alpha, beta, and Mach sweeps are obtained from CFD or
wind-tunnel data. Prandtl-Glauert correction is a fallback for Mach
extrapolation only. All geometry in SI (metres). Coefficients corrected to jig.
See `doc/loads_aero_db.md` for the full column schema, file naming convention,
interpolation method, and Mach extrapolation policy.

### 1b. Condition list
Fields required at minimum: condition ID, description, category (maneuver/gust/ground),
altitude, EAS, gross weight, CG, load factor or gust velocity, active control surfaces.
Options: JSON with schema; CSV with fixed column order.

**Direction:** [x] CSV format; generated by an external workflow and consumed
read-only by this program. Two separate structural columns (`category` and
`maneuver_type`) distinguish the top-level load category from the specific case
type. All physical quantities in SI. Control surface deflection columns are one
per surface (`_deg`); blank or omitted cells equal 0 deg and are converted to
radians at ingestion in `condition.py`.

### 1c. Loads Reference Axis (LRA) definition
Each entry defines a spanwise station, its xyz position, and the local frame orientation.
Options: CSV columns (y_ft, x_ft, z_ft, orientation); JSON array.

**Direction:** [x] JSON format; one file per lifting surface placed in
`data/lra/`.  Each file defines an ordered array of stations along the
Loads Reference Axis spine for that surface.  All coordinates are in SI
(metres), referenced to the global aircraft frame (origin at fuselage
station zero / waterline zero / buttline zero).

Each station entry requires three fields:

| Field | Type | Unit | Notes |
|---|---|---|---|
| `station_id` | string | — | Unique label, e.g. `"WS0.000"` (Wing Station at y = 0 m) |
| `position_m` | float[3] | m | Global-frame coordinates `[x_m, y_m, z_m]` of the LRA reference point at this cut |
| `normal_nd` | float[3] | — | Unit vector pointing in the direction of **positive Cn** (positive section normal force); magnitude must equal 1.0 ± 1 × 10⁻⁶ |

Conventions:
- For a wing, `normal_nd` points approximately in the dorsal direction
  (positive global z for a body-axes system where z is up).
- For the vertical tail, `normal_nd` points approximately in the
  positive-y (starboard) direction.
- Stations must be listed in order of increasing spanwise `y_m`.
- A chord-direction vector is **not** stored here; chordwise direction
  at each strip station is inferred from the aero database geometry
  (`c_m` and adjacent `y_m` values in the baseline CSV).  Moment
  transfer from the strip aerodynamic centre to the LRA reference
  point uses the full 3-D offset between the strip c/4 position and
  `position_m`.

File naming: `lra_<surface>.json` (e.g. `lra_wing.json`,
`lra_htail.json`, `lra_vtail.json`).

See `doc/lra_db.md` for the complete JSON schema, validation rules, and
`lra.py` parsing requirements.

### 1d. Beam stiffness distribution (for aeroelastic corrections)
Columns needed: spanwise station y_ft, EI_lbf_ft2, GJ_lbf_ft2, EA_lbf.
Options: CSV; same file as LRA; JSON.

**Direction:** [x] NASTRAN BDF format. The `sbeam` library
(`/Users/seanomeara/Documents/99-Tests/sbeam`) is integrated as the beam solver.
A new module `src/beam.py` owns all beam model I/O and structural dynamics
computation. `sbeam` is a dependency (installed editable via `pip install -e`
from its project directory).

**Preferred interface — modal reduced-order model (ROM):**

`src/beam.py` calls the sbeam API in three steps:

1. **Parse** — `sbeam.parser.bdf_reader.parse_bdf(path)` reads the BDF file
   (GRID, CBAR, PBAR, MAT1, CONM2, SPC, EIGRL cards) into a `BulkData` object.
   Units in the BDF must be SI (metres, kg, N); if an imperial BDF is supplied it
   is converted to SI immediately at ingestion using `src/unit_convert.py` before
   any sbeam assembler is called.

2. **Assemble** — call `sbeam.assembly.stiffness.assemble_global_stiffness(bulk)`
   → K (6N × 6N ndarray) and
   `sbeam.assembly.mass_matrix.assemble_global_mass(bulk)` → M (6N × 6N ndarray).

3. **Modal solve** — call `sbeam.solver.sol103.solve(bulk, case_control)` →
   `Sol103Result` containing:
   - `mode_shapes` Φ — ndarray (n_dofs × n_modes), mass-normalised so Φᵀ M Φ = I
   - `eigenvalues` ω² — ndarray (n_modes,) in rad²/s²
   - `frequencies_hz` — ndarray (n_modes,)

   From these, `beam.py` forms the modal-reduced system:

   | Quantity | Expression | Shape |
   |---|---|---|
   | Modal stiffness | K_modal = diag(ω²) | (n_modes × n_modes) diagonal |
   | Modal mass | M_modal = I | (n_modes × n_modes) identity |
   | Modal damping | D_modal = diag(2 ζᵢ ωᵢ) | (n_modes × n_modes) diagonal |

   Modal damping ratios ζᵢ (dimensionless) are supplied by the user via
   `config/defaults.json` (a single structural damping ratio applied to all
   modes, or a per-mode array). Since sbeam Phase 1 does not assemble a
   physical damping matrix, modal damping is the only supported approach.

**LRA station response recovery:**

`beam.py` maintains a DOF index map that associates each LRA station (from the
JSON file defined in decision 1c) with the sbeam grid node nearest to that
station's `position_m[1]` (spanwise y). The slice of Φ at those DOF rows gives
the modal displacement/rotation influence at each LRA cut. This slice, together
with the modal coordinates q solved for a given aero load vector, recovers
physical deflection u at the LRA spine:

  u_LRA = Φ_LRA × q

where Φ_LRA is the (n_lra_dofs × n_modes) sub-matrix of Φ.

**Full K, M fallback (optional):**

For static aeroelastic correction (flexible camber line, jig-shape
determination) the full K and M matrices are available directly from the
assemblers without a modal solve. `aeroelastic.py` may call `beam.get_K()` and
`beam.get_M()` when a direct stiffness solution is preferred over modal
superposition.

**Flexibility matrix (see also decision 1e):**

The modal truncated flexibility matrix is:

  F_flex = Φ Λ⁻¹ Φᵀ   where Λ = diag(ω²)

This is computed on demand by `beam.py` and eliminates the need to supply an
external flexibility matrix (decision 1e) unless a pre-computed NASTRAN DMIG
flexibility is explicitly required by the project.

**New config keys required (see decision 14):**
- `beam_bdf_path` — path to the structural BDF file (one per lifting surface or
  a single aircraft-level file; TBD per decision 5)
- `beam_n_modes` — number of modes to retain in the ROM (suggested default: 20)
- `beam_zeta` — structural damping ratio(s); scalar or list of length n_modes
  (suggested default: 0.02)

### 1e. Flexibility matrix (pre-computed AIC/flexibility matrix input)
An alternative to beam stiffness — externally generated from a NASTRAN or FEM solve.
Options: dense CSV (n_stations × n_stations); NASTRAN DMIG card subset.

**Direction:** [x] Derived internally from the sbeam modal ROM (see decision 1d):
`F_flex = Φ Λ⁻¹ Φᵀ`. An external pre-computed flexibility matrix is not
required for this program. If a user wishes to supply an externally generated
NASTRAN DMIG flexibility, that is deferred to a future enhancement and is out
of scope for the initial release.

## 2. Continuous Turbulence (PSD Gust) — FAR 25.341(b)

FAR 25.341(b) requires PSD gust analysis. This needs frequency response functions
(FRF) of the aircraft structure. Strip-theory aerodynamics alone are insufficient;
DLM-based aerodynamics are the industry standard for this load path.

**Option A — Accept externally computed FRFs as an input.**
User supplies FRF tables (load vs. frequency) from an external DLM/NASTRAN run.
This program applies the PSD integral and reports limit loads. Low implementation
risk; shifts the DLM dependency to the user.

**Option B — Simplified 2-DOF plunge-pitch model (self-contained).**
Implement a rigid-body plunge-pitch FRF using the strip-theory aero database.
Approximate; suitable for early-design sizing. Not acceptable for certification
without external validation.

**Option C — Defer PSD gust (out of scope for this program).**
Mark FAR 25.341(b) as requiring an external tool. Implement only discrete
1-cosine gust per FAR 25.341(a). State explicitly in the spec.

**Direction:** [x]

**Phase 1 (this release) — two separate methods, both self-contained:**

*Discrete gust — static equivalent method (pre-Amendment 25-86):*
Implement the static equivalent gust formula from Lomax §4 and the original
FAR Part 25 Appendix G (before Amendment 25-86, 1996). Design gust velocity
`u_gust_m_s` is taken from the pre-1996 regulatory table (50 fps EAS / 15.24 m/s
at sea level; 25 fps EAS / 7.62 m/s at 20 000 ft; linear interpolation between
altitudes). There is no 1-cosine profile and no sweep over gust gradient H.
The gust alleviation factor `k_gust_nd` uses the same mass ratio formula as the
current regulation. The incremental load factor `delta_nz_nd` is computed with
the static equivalent formula (see `doc/analysis_code.md §b2`). This is NOT the
modern Tuned Discrete Gust (TDG); it is an earlier, simpler approach adequate for
early-design sizing. Regulatory basis: original FAR 25 Appendix G (pre-Amendment
25-86). The config keys `gust_gradient_min_ft`, `gust_gradient_max_ft`, and
`n_gust_steps` are reserved for Phase 2 and unused in Phase 1.

*Continuous turbulence — simplified 2-DOF rigid-body frequency response (Option B):*
Implement a self-contained rigid-body plunge-pitch frequency response model using
strip-theory aerodynamics (no DLM, no NASTRAN required). The model computes
complex FRFs H_nz(jω) and H_My(jω) for vertical load factor and wing root bending
moment, integrates against the Von Kármán turbulence PSD per FAR 25.341(b) and
AC 25.341-1, and derives RMS section loads σ_load. Design limit loads are
`k_sigma_nd × σ_load` where `k_sigma_nd = 3.0` (limit load factor method per
AC 25.341-1). Suitable for early-design sizing; external DLM validation required
before certification. See `doc/analysis_code.md §b3`.

**Phase 2 (deferred — future release):**
Full NASTRAN DLM or ZONA51 analysis for proper PSD continuous turbulence FRFs
and the 1-cosine Tuned Discrete Gust (TDG) per current FAR 25.341(a) and
AC 25.341-1. Requires a separate DLM aerodynamic model and a modal structural FEM
result. The condition list `maneuver_type` values `discrete_gust_vertical`,
`discrete_gust_lateral`, and `continuous_turbulence` remain valid for Phase 2
routing once that code path exists.

## 3. Ground Loads — Landing Gear Model

Ground load cases (FAR 25.473–25.511) require landing gear characteristics
(stroke, spring rate, damping) to compute dynamic reactions. Two approaches:

**Option A — Quasi-static (no gear model required).**
Use design sink rate V_sink = 10 fps and assumed gear efficiency η = 0.80
per FAR 25.473(b) to compute a static vertical reaction. Standard for initial
structural sizing. Gear stroke `d_stroke_ft` and efficiency `eta_gear` become
user inputs per condition.

**Option B — Dynamic gear model.**
Add landing gear spring/damper as an additional input. Integrate gear stroke
during landing impact to compute time-varying reaction. Higher fidelity;
significantly more complex.

**Direction:** [x]

**Phase 1 (this release) — quasi-static reserve energy method (Option A):**
Use the FAR 25.473 reserve energy formula to compute the peak vertical gear
reaction from the design sink rate and gear stroke. No gear spring/damper model
required. `eta_gear_nd` (gear absorption efficiency, default 0.80 per
FAR 25.473(b)) and `d_stroke_m` (available gear stroke) are user-supplied per
condition or from `config/defaults.json`. The peak reaction is applied as a
quasi-static vertical load at the gear attachment point; the effective load
factor `nz_eff_nd = f_gear_n / w_aircraft_n` is passed to `loads.py`. All
landing sub-cases (level, tail-down, one-gear, lateral drift, rebound) are
implemented using the quasi-static gear reaction. Static ground handling cases
(braked roll, ground turn, nose-wheel yaw, towing, pivoting, jacking) are also
quasi-static and require no gear model. See `doc/analysis_code.md §c` and `§d`.

**Phase 2 (deferred — future release):**
Dynamic impact analysis with a spring-damper gear model per `doc/analysis_code.md
§d3`. Requires gear spring stiffness `k_gear_n_m` and damping `c_gear_ns_m` as
additional inputs. Integrates the coupled aircraft + gear equations of motion to
produce a time-history of gear attachment loads. Phase 2 replaces `§d1` for
landing conditions only; static ground handling (§c) remains quasi-static.

## 4. Coordinate System and Sign Convention

No axis system is defined. Mixed conventions are a common source of sign errors
in loads programs.

**Option A — Body axes (x forward, y starboard, z down).**
Consistent with Lomax, Wright & Cooper, and FAR/CS texts. Standard for
flight mechanics and structural loads reporting.

**Option B — NASTRAN basic axes (x forward, y port, z up).**
Standard in the FEM community; avoids sign transformation when comparing
to NASTRAN model output.

A single convention must be chosen, documented in `doc/analysis_code.md`,
and applied consistently to all strip integration and inertia summation.

**Direction:** [x]

**Global structural coordinate frame (used for all positions, LRA, beam model,
and section-load reporting):**
- **x**: positive **AFT** (fuselage stations increase from nose to tail)
- **y**: positive **STARBOARD** (butt lines increase from centerline outboard to right)
- **z**: positive **UP** (waterlines increase upward)
- **Origin**: aircraft datum — FS 0 / BL 0 / WL 0

This follows the aircraft structural convention (fuselage stations / butt lines /
waterlines) used in NASTRAN aircraft structural models.

**Aerodynamic quantities** (CL, CD, CM, Cn, Cl, CY, α, β, p, q, r, all control
deflections) retain the **standard aerodynamic body-axis sign convention**:
x forward, y starboard, z down — per Anderson and Etkin. These quantities are in
the aerodynamic body frame, NOT the structural frame, and must be explicitly
transformed when used in structural load computations.

**Section loads (SMT) at each LRA cut** follow the **Lomax §5 sign convention**
(*Structural Loads Analysis for Commercial Transport Aircraft*, AIAA 1996). For a
wing cut at y = y_m with the outboard free body (face normal = −ŷ):

| Quantity | Variable | Positive direction | Notes |
|---|---|---|---|
| Vertical shear | `vz_n` | +z (upward) | Positive under lift loading |
| Chordwise shear | `vx_n` | +x (aft) | Positive under drag loading |
| Spanwise axial | `fy_n` | +y (outboard) | Spanwise tension |
| Out-of-plane bending | `mx_nm` | Upward bending | Upper surface in compression; Lomax positive |
| Torsion | `my_nm` | Leading edge up | Nose-up twist; Lomax positive torsion |
| In-plane bending | `mz_nm` | +x displacement (aft) | Chordwise bending |

The full coordinate system, sign conventions, and variable naming rules are
documented in `doc/variable_definition.md` and `doc/aerospace_variables_reference.csv`.

## 5. Multi-Component LRA — Per-Surface or Single Spine

Industry programs define a separate LRA for each major structure: wing LRA,
horizontal tail LRA, vertical tail LRA, fuselage spine. Each has its own
station cuts and section-load outputs.

**Option A — Per-surface LRAs (industry standard).**
Separate section cuts and output tables for wing, HT, VT, fuselage.
Matches how structural sizing is organized.

**Option B — Single unified LRA.**
One set of stations along the aircraft centerline/spine.
Simpler to implement; less useful for surface-by-surface structural sizing.

**Direction:** [ x ] Option A — Per-surface LRAs.

User-supplied JSON files define each surface LRA.  The LRA need not be
straight — kinks (e.g. winglet junctions) are supported by specifying 3-D
`position_m` coordinates and a per-station `normal_nd` that rotates to match
the local surface orientation.  Station ordering is defined by the user in the
JSON file (spine index order); `lra.py` does not enforce a single monotone
spatial axis.  See `doc/lra_db.md` for the updated file format.

is currently written for FAR 25 only. FAR 23 has different n_z envelopes
(normal/utility/acrobatic categories), different gust criteria, and a
simplified maneuver load approach.

**Option A — FAR 23 as a first-class parallel implementation.**
Separate condition categories with FAR 23-specific n_z formulas (FAR 23.337)
and gust velocities (FAR 23.341).

**Option B — Category-agnostic core; user supplies compliant values.**
The program does not enforce FAR 23 formulas. The condition list contains
user-specified n_z and gust velocities that the engineer has pre-computed
per the applicable regulation.

**Option C — FAR 23 deferred; FAR 25 only for initial release.**

## 16. Gust Alleviation Factor K_g — Formula Source

`K_g` was added to the symbol table but the formula is not stated.
FAR 25.341(a)(4) defines:

```
K_g = 0.88 × μ_g / (5.3 + μ_g)
```

where the mass ratio `μ_g = 2m / (ρ × c̄ × S × a_slope)` and `a_slope` is
the wing lift curve slope (per radian).

## 25. Load Combination Rules for Asymmetric Cases

The spec states rolling pull-out is "combined roll rate + symmetric pull-up"
but does not define the combination rule.

**Option A — Direct superposition.**
Section loads from roll rate (antisymmetric) and pull-up (symmetric) are added
algebraically at each station.

**Option B — SRSS (square root of sum of squares).**
Used when the two components are not correlated in phase.

**Option C — Envelope of direct sum and each component alone.**
Report the worst of: (roll alone), (pull-up alone), (roll + pull-up combined).

Industry standard for rolling pull-out is direct superposition (Option A)
per FAR 25.349. Confirm this applies.
